1 نظرة معمارية على الـ Stack

المشروع مبني على معمارية Three-Tier Client-Server كلاسيكية: طبقة عرض في المتصفح، طبقة منطق على الخادم، وطبقة تخزين في قاعدة البيانات. كل طبقة منعزلة ومسؤولياتها واضحة.

┌──────────────────────────────────────────────────────────────────────┐ │ BROWSER (Presentation Tier) │ │ HTML5 · CSS3 · Vanilla ES2020 │ │ - DOM rendering · fetch() · localStorage (JWT) │ └────────────────────┬─────────────────────────────────────────────────┘ │ HTTP/JSON (REST) + Authorization: Bearer ▼ ┌──────────────────────────────────────────────────────────────────────┐ │ EXPRESS SERVER (Application Tier) — Node.js · TypeScript │ │ Routes → Middleware (requireAuth) → Controllers → Services → │ │ → Repositories → Mongoose ODM │ └────────────────────┬─────────────────────────────────────────────────┘ │ BSON (Mongo Wire Protocol) ▼ ┌──────────────────────────────────────────────────────────────────────┐ │ MONGODB (Data Tier) — Document Store · Indexed Collections │ │ users · transactions · budgets │ └──────────────────────────────────────────────────────────────────────┘

١-١. الـ Presentation Tier — المتصفح

المتصفح هنا مش مجرد viewer — ده runtime كامل. كل صفحة HTML فيها structure، الـCSS بيتحكم في الـvisual layer، والـJavaScript بيتعامل مع الـDOM كـ tree of nodes ويعدّل عليه in-place.

  • HTML5: بيوفّر semantic elements زي <header>, <main>, <aside>, <section> — مهمة للـaccessibility وللـSEO، ومش مجرد decoration.
  • CSS3: بنستخدم custom properties (المعروفة بـ CSS variables) في :root، Flexbox + CSS Grid للـlayout، وmedia queries للـresponsive design.
  • Vanilla JavaScript: من غير framework (لا React ولا Vue) — اعتمدنا على native DOM APIs: document.getElementById, addEventListener, fetch, وevent delegation.
ليه Vanilla مش framework؟

المشروع academic ومحدود الـscope — استخدام framework هيضيف build pipeline، virtual DOM overhead، وlearning curve بدون فايدة حقيقية. الـVanilla بيخلّيك فاهم تحت الكبوت بيحصل إيه — وده اللي اللجنة بتسأل عنه.

١-٢. الـ Application Tier — الـ Backend

الـBackend مبني بـ Node.js + Express + TypeScript. Node بيدّينا single-threaded event loop مع non-blocking I/O، وده اللي يخلّيه ينفع لتطبيق فيه DB calls كتيرة. Express هو minimalist routing layer فوق HTTP module بتاع Node. TypeScript يدّينا static typing + compile-time safety.

التقسيم الـ Layered داخل السيرفر:

Routes Middleware Controllers Services Repositories Models

كل طبقة مسؤوليتها واضحة:

  • Routes: بتربط الـURL pattern بالـhandler function.
  • Middleware: بيشتغل قبل الـhandler — في حالتنا الـJWT verification.
  • Controllers: HTTP-aware — بيقرأوا الـrequest ويرجّعوا response.
  • Services: فيها الـbusiness logic مفصولة عن HTTP — تقدر تستخدمها من CLI أو cron job بنفس السهولة.
  • Repositories: data access layer — بتغلّف الـMongoose queries.
  • Models: schemas + validation.

١-٣. الـ Data Tier — MongoDB

المشروع بيستخدم MongoDB اللي هي document-oriented NoSQL database — مش relational. كل document هو BSON object (نسخة binary من JSON). نتعامل معاها من خلال Mongoose — اللي هي ODM (Object Document Mapper) — نظير الـORM في عالم الـSQL.

Schema definition with constraints — src/models/Transaction.ts 
const TransactionSchema = new Schema<Transaction>({
  userId:   { type: Schema.Types.ObjectId, ref: 'User', required: true, index: true },
  type:     { type: String, required: true, enum: ['income', 'expense'], index: true },
  amount:   { type: Number, required: true, min: 0.01, max: 9999999.99 },
  category: { type: String, required: true, trim: true, maxlength: 50 },
  dateTime: { type: Date,   required: true, default: () => new Date() },
  description: { type: String, trim: true, maxlength: 200 },
}, { timestamps: true });

TransactionSchema.index({ userId: 1, dateTime: -1 });   // compound index
ملاحظة مهمة — Relational vs Document

في SQL لو عندك Users وTransactions، هتعمل FOREIGN KEY + JOIN. في MongoDB ممكن إما تعمل referencing (زي اللي عملناه بـ userId: ObjectId) أو embedding. اختيارنا للـreferencing لأن الـtransactions ممكن تكبر جداً والـembedded array هتوصل لحدود الـ16 MB document limit.

2 المفاهيم الـTechnical الأساسية

المفاهيم اللي مكتوبة في المشروع — مش بس syntax، ده engineering decisions ليها trade-offs.

٢-١. CRUD Operations

الـCRUD هو Create / Read / Update / Delete. في الـREST API بنعمل mapping بينها وبين الـHTTP verbs:

VerbEndpointالوظيفة
POST/api/v1/finance/transactionsإنشاء transaction جديدة
GET/api/v1/finance/transactions?q=...قراءة مع filter/sort/paginate
PUT/api/v1/finance/transactions/:idتحديث (replace)

ملاحظة: PUT ضد PATCH — الـPUT بيستبدل الـresource بالكامل (idempotent)، أما PATCH بيعدّل جزء منه. مشروعنا بيستخدم PUT مع partial validation عشان السهولة.

٢-٢. Design Patterns المستخدمة

(أ) Singleton Pattern

كل الـrepositories + FinanceManager متنفّذين كـ Singletons — يعني instance واحدة فقط طول حياة الـprocess. الفائدة: مفيش redundant connections ولا state duplication.

Singleton in TypeScript 
export class TransactionRepo {
  private static instance: TransactionRepo | null = null;
  private constructor() {}      // ← مفيش حد يقدر يعمل new من بره

  static getInstance(): TransactionRepo {
    if (!TransactionRepo.instance) {
      TransactionRepo.instance = new TransactionRepo();
    }
    return TransactionRepo.instance;
  }
}

(ب) Observer Pattern

الـFinanceManager بيـextend كلاس Subject اللي عنده attach / detach / notify. لما تضاف transaction، الـAlertSystem (الـObserver) بيتعمله notify ويسجّل alert. ده decoupling — الـFinanceManager مش لازم يعرف مين بيسمعه.

(ج) Strategy Pattern

التصدير (export) لـ CSV/PDF/Chart متنفّذ كـ interchangeable strategies بتطبق نفس الـinterface (ExportStrategy). تقدر تضيف JsonExport بكره من غير ما تغير في الـcontroller.

(د) Repository Pattern

كل model ليه Repository بيغلّف الـDB queries. ده بيخلّي الـbusiness logic مش متعلّقة بالـDB engine — لو قررنا نهاجر لـ PostgreSQL بكره، التغيير هيكون في الـRepositories بس.

٢-٤. Authentication & Middleware

الـauthentication بنعملها بـ JWT (JSON Web Token). الـtoken بيتولد بعد login، بيتخزّن في الـlocalStorage، وبيتبعت في كل request في الـAuthorization header كـ Bearer <token>.

requireAuth Middleware 
export function requireAuth(req, res, next) {
  const header = req.header('Authorization');
  const token  = header?.startsWith('Bearer ') ? header.slice(7) : null;
  if (!token) return res.status(401).json({ error: 'Missing token' });
  try {
    const decoded = jwt.verify(token, CONFIG.JWT_SECRET);
    req.user = { userId: decoded.userId, role: decoded.role };
    return next();   // ← يكمّل للـ controller
  } catch {
    return res.status(401).json({ error: 'Invalid or expired token' });
  }
}
الفرق بين Authentication وAuthorization

Authentication = أنت مين؟ (verify identity)
Authorization = مسموحلك تعمل إيه؟ (verify permissions)
في المشروع، requireAuth بيعمل authN، وrequireRole('admin') بيعمل authZ.

٢-٥. State Management

السيرفر stateless بالكامل — مفيش session مخزّنة. كل request بيحمل JWT فيه الـidentity. ده بيدّينا horizontal scalability — أي عدد من الـserver instances ممكن يخدموا أي request.

في الـfrontend، الـstate بسيط:

  • Persistent state: localStorage.token
  • Page-level state: closure variable داخل الـIIFE (زي state.q, state.page في transactions.html)
  • Server-derived state: بنـre-fetch بعد كل عملية ناجحة عشان نضمن consistency

٢-٦. Asynchronous Programming

كل I/O operation (DB query, fetch, fs) non-blocking. بنستخدم async/await فوق الـPromises عشان الكود يبقى linear ومش callback hell.

Sequential vs Parallel async operations 
// ❌ Sequential — أبطأ، لأن كل وحدة بتستنى اللي قبلها
const summary  = await apiFetch('/reports/summary');
const txData   = await apiFetch('/finance/transactions');
const budgets  = await apiFetch('/budgets');

// ✅ Parallel — أسرع، لأنهم بيتنفّذوا في نفس الوقت
const [summary, txData, budgets] = await Promise.all([
  apiFetch('/reports/summary'),
  apiFetch('/finance/transactions'),
  apiFetch('/budgets'),
]);

3 الـUI–Backend Integration بالتفصيل

إزاي click على زرار في المتصفح بيتحوّل لـ DB write وبيرجع UI update؟ الـflow كامل.

٣-١. الـRequest Lifecycle — مثال عملي

تخيّل المستخدم ضغط على "+ Add Transaction":

  1. DOM event: الـclick event بيـtrigger الـhandler اللي اتسجل بـ addEventListener.
  2. Client-side validation: بنتأكد إن الـamount > 0 والـcategory مختارة. لو فيه مشكلة، بنحطّ has-error class على الـfield ونـreturn.
  3. fetch() call: apiFetch بيبعت POST request فيها Authorization: Bearer ... + JSON body.
  4. Express routing: الـrouter بيلاقي POST /transactions ويبعتها للـmiddleware chain.
  5. Middleware: requireAuth بيـverify الـJWT ويحطّ req.user.
  6. Controller: createTransaction بيـvalidate الـbody، يستدعي finance.addTransaction، ويرجّع JSON.
  7. Service / Repository: FinanceManager بيـnotify الـobservers وبعدين TransactionRepo بيـcreate الـdocument في MongoDB.
  8. Response: السيرفر بيرجّع 201 Created + normalized object + new balance.
  9. UI update: الـhandler بيقفل الـmodal، يـshow toast، ويعمل loadDashboard() عشان يـre-fetch ويـre-render.

٣-٢. الـDOM Rendering Strategy

بدل ما نـmanipulate كل node لوحده، بنبني HTML string كامل ونحطّه في innerHTML. ده simpler لكن لازم نـescape أي user input عشان نمنع XSS.

XSS protection via escapeHtml 
function escapeHtml(s) {
  return String(s ?? '').replace(/[&<>"']/g, c => ({
    '&': '&amp;', '<': '&lt;', '>': '&gt;',
    '"': '&quot;', "'": '&#39;',
  }[c]));
}

// في رندر الجدول:
tbody.innerHTML = items.map(tx => `
  <tr>
    <td>${escapeHtml(tx.description)}</td>  <!-- مهم! -->
    <td>${escapeHtml(tx.category)}</td>
  </tr>`).join('');

٣-٣. Event Delegation

بدل ما نسجّل onclick لكل صف في الجدول، بنسجّل single listener على الـtbody ونستخدم e.target.closest('.js-delete'). ده بيوفّر memory ويشتغل مع الـelements اللي بتتضاف dynamically بعدين.

٣-٤. Server-Side Search/Filter/Sort/Paginate

الـfilter builder هو single source of truth — الـlist، الـcount، والـCSV export كلهم بيستخدموه:

buildFilter(q: TransactionQuery): FilterQuery {
  const filter: any = { userId: new ObjectId(q.userId) };
  if (q.type) filter.type = q.type;
  if (q.category) filter.category = q.category;
  if (q.from || q.to) {
    filter.dateTime = {};
    if (q.from) filter.dateTime.$gte = q.from;
    if (q.to)   filter.dateTime.$lte = q.to;
  }
  if (q.q?.trim()) {
    const escaped = q.q.trim().replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
    filter.description = { $regex: escaped, $options: 'i' };
  }
  return filter;
}

4 جلسة Viva — ٢٠ سؤال صعب

دي الأسئلة اللي الـTA هيسألهالك في الـdiscussion. الإجابات concise ومركّزة على الفهم العميق.

1SQL Injection ممكن تحصل في المشروع؟

المشروع بيستخدم MongoDB مش SQL، فالـclassical SQL Injection مش applicable. بس فيه نظيرها — NoSQL Injection — اللي بتحصل لو حد بعت JSON فيه operator زي {"email": {"$gt": ""}} بدل string. الـMongoose schema بيحمينا لأنه بيعمل type casting — لو الـschema بيقول email: String والمستخدم بعت object، Mongoose هيرفضه. كمان ObjectId casting في الـmiddleware بيمنع تمرير invalid IDs.

2لو الـproject كان SQL، كنت هتحمي إزاي من SQL Injection؟

الإجابة الوحيدة الصحيحة: parameterized queries / prepared statements. مينفعش أبداً نـconcatenate الـuser input في الـSQL string. مثال:

// ❌ خطر — vulnerable
db.query(`SELECT * FROM users WHERE email = '${email}'`);

// ✅ آمن — driver بيـ escape القيم
db.query('SELECT * FROM users WHERE email = ?', [email]);

كمان least-privilege DB user, stored procedures, وORM-level type checking بيضيفوا طبقات حماية.

3ليه bcrypt مش SHA-256 للـ password hashing؟

SHA-256 سريع — وده عيبه هنا. الـattacker اللي يـleak الـDB يقدر يجرّب مليارات الـcandidates في الثانية على GPU. bcrypt مصمّم يكون slow + adaptive — عنده cost factor (في كودنا 10) يعني 2^10 iterations. كمان bcrypt بيـsalt automatically — كل password ليه unique hash حتى لو نفس الـplaintext، وده بيقتل الـrainbow table attacks.

4إيه اللي جوّه الـJWT، وممكن أعدّل عليه؟

الـJWT = ثلاث أجزاء مفصولة بـ .: Header (algorithm) · Payload (claims) · Signature (HMAC). كله Base64URL-encoded مش encrypted — أي حد يقدر يقرا الـpayload. لكن لو حد عدّل أي حرف، الـsignature هتبقى invalid لأن السيرفر هيـrecompute الـHMAC بـ JWT_SECRET وهيلاقي اختلاف.

القاعدة: ما تحطّش sensitive data في الـpayload — لأنها readable.

5إيه ترتيب تنفيذ الـmiddleware؟ ولو حد نسى يكتب next() هيحصل إيه؟

الـmiddleware بيتنفّذ بترتيب التسجيل في app.use() / router.use(). كل واحدة بتاخد (req, res, next)، ولازم تستدعي next() عشان الـrequest يكمّل. لو نسيت — الـrequest هيـhang لحد ما الـclient يعمل timeout، أو لو الـresponse اتبعتت داخل الـmiddleware (res.status(401).json(...)) فالـrequest هيخلص هناك من غير ما يكمّل.

6إيه فايدة الـcompound index (userId, dateTime) اللي عملناه؟

أكتر query بنعملها هي "هاتلي transactions الـuser ده مرتّبين بالتاريخ". الـcompound index بيسمح للـDB engine إنه يـmatch الـuser ويـscan النتايج already-sorted في الـB-Tree — من غير in-memory sort. النتيجة: O(log n) بدل O(n log n). الترتيب مهم — {userId: 1, dateTime: -1} مش زي {dateTime: -1, userId: 1} — الـprefix rule.

7اشرح CSS Specificity ولو عندي inline style + class rule مين هيكسب؟

الـspecificity بتتحسب كـ 4-tuple: (inline, id, class, tag). الترتيب من الأقوى للأضعف:

  1. !important (يكسر كل القواعد)
  2. Inline style = (1,0,0,0)
  3. ID selector = (0,1,0,0)
  4. Class / attribute / pseudo-class = (0,0,1,0)
  5. Element / pseudo-element = (0,0,0,1)

فلو فيه style="color:red" على عنصر و.btn { color: blue } في الـstylesheet، الـinline هتكسب. وعشان كده عملنا refactor للأزرار في المشروع من inline لـ classes — عشان نقدر نـoverride بسهولة.

8ليه الـSingleton في TypeScript آمن أكتر من JS؟

الـprivate constructor + الـaccess modifier بيتحقّقوا at compile-time. في JS لو حد كتب new TransactionRepo() من بره، مفيش حاجة هتمنعه. في TS، الـcompiler هيرمي error. كمان static instance + الـmodule caching في Node بيضمنوا إن الـfile بيتـload مرة واحدة فقط.

9Pagination بـ skip() + limit() — متى تبقى مشكلة؟

الـskip(n) بيمشي على n documents أولاً قبل ما يبدأ يرجّع. لو الـpage رقم 1000 على collection فيها مليون document، الـDB هتعدّي على 25,000 doc من غير فايدة. الحل الـproduction-grade هو cursor-based pagination: بدل ?page=N، تستخدم ?after=<last_id> وتعمل find({ _id: { $gt: last_id } }) — ده O(log n) بفضل الـindex. مشروعنا بيستخدم offset pagination لأن البيانات صغيرة وده الـstandard academic approach.

10إزاي بنمنع XSS في الـdynamically-rendered tables؟

أي user-provided string (description, category) بيتمر على escapeHtml() اللي بيحوّل <, >, ", ', & لـ HTML entities. كده لو واحد كتب <script>alert(1)</script> في الـdescription، هيتـrender كـ text مش كـ code. كمان لازم نتجنّب innerHTML للـdata اللي مش متأكدين منها — في الحالات الحرجة نستخدم textContent.

11ليه عملنا debounce على الـsearch input ومش trigger كل keystroke؟

بدون debounce: كل حرف يدوس عليه المستخدم = HTTP request = DB query. لو كتب "transport" يبقى 9 requests منهم 8 مش مهمين. بـ debounce(300ms): الـhandler بيستنى لحد ما المستخدم يهدا 300ms من غير كتابة، وبعدها يبعت request واحد. ده بيوفّر network, DB load, وrender cycles.

12Error handling في الـlayered architecture — مين بيـcatch إيه؟

عندنا تلات طبقات:

  1. Validation layer: في الـcontroller قبل ما نـtouch الـDB. ترجّع 400 Bad Request.
  2. Domain layer: في الـservice (مثلاً assertValidAmount) — بترمي Error object.
  3. Persistence layer: Mongoose errors (ValidationError, duplicate key 11000). بنـcatch في الـcontroller ونـmap لـ 409 Conflict أو 400.

أخر طبقة هي process.on('unhandledRejection', ...) في server.ts — ده safety net عشان أي async error اتفلت ما يكرّش الـprocess.

13ليه async/await أحسن من callbacks؟

تلات أسباب:

  • Linear control flow: الكود يقرا من فوق لتحت زي الـsynchronous code، مفيش callback pyramid.
  • Error propagation: try/catch بيشتغل عادي. مع الـcallbacks كنت تضطر تـhandle الـerr في كل callback.
  • Composability: Promise.all, Promise.race, Promise.allSettled بتدّيك orchestration primitives ما كنتش متاحة بسهولة في الـcallbacks.

تحت الكبوت: async function بترجع Promise، وawait بيـpause الـexecution ويرجّع التحكم للـevent loop لحد ما الـPromise يـresolve.

14ليه استخدمنا TypeScript ومش JavaScript؟

TypeScript = JS + static type system. الفوايد:

  • Compile-time errors: typos وwrong types بتتمسك قبل الـruntime.
  • Better tooling: autocomplete, refactoring, jump-to-definition.
  • Self-documenting: الـinterface ذاتها contract بين الأكواد.

التكلفة: build step (tsc أو ts-node) + وقت تعلم. لكن في مشروع متوسط أو كبير، الـsavings في الـbugs أكتر من التكلفة بكتير.

15إزاي تخلّي الـapp ده يستحمّل ١٠٠ ألف user؟ (Performance / Scaling)

الحلول مرتبة من الأرخص للأغلى:

  1. DB indexing: تأكد إن كل query path مغطّى بـ index.
  2. Connection pooling: Mongoose بيعمله by default — تأكد من الحجم.
  3. Caching layer: Redis للـread-heavy endpoints زي /reports/summary.
  4. Horizontal scaling: لأن السيرفر stateless، تقدر تشغّل ٥ instances ورا load balancer.
  5. Read replicas في MongoDB + sharding لو دي مش كفاية.
  6. CDN للـstatic assets (HTML/CSS/JS).
16ليه ما خزّناش الـJWT في httpOnly cookie بدل localStorage؟

الإجابة الـacademic: localStorage عرضة لـ XSS — أي injected script يقدر يقراه. httpOnly cookie الـJS ميقدرش يـreadه — أأمن. بس الـcookie بيـexpose الـapp لـ CSRF — فلازم CSRF token أو SameSite=Strict. في مشروع تعليمي مع strict XSS prevention + CORS off، الـlocalStorage مقبول. في production، الـcookie أحسن.

17إيه الفرق بين الـNormalization في SQL والـdocument modeling في MongoDB؟

في SQL بنطبّق 1NF, 2NF, 3NF لتقليل الـredundancy — كل entity في table لوحدها وJOIN بيـreconstruct العلاقات. في MongoDB القاعدة عكسية: design for the query. لو الـdata دايماً بتتقرا مع بعض، اعمل embedding. لو منفصلة أو الـchild بيكبر بلا حدود، اعمل referencing. مشروعنا اختار referencing للـtransactions لأنها قابلة للنمو.

18إزاي بنضمن إن user A ميقدرش يـdelete transaction لـ user B؟

على مستويين:

  1. Middleware: requireAuth بيـverify الـJWT ويـextract req.user.userId.
  2. Repository query: استخدمنا findOneAndDelete({ _id, userId }) بدل findByIdAndDelete(id). لو الـid موجود بس مش بتاع الـuser، الـquery هترجّع null والـcontroller هيرجّع 404.

ده النمط المعروف بـ ownership check at query level — أأمن من fetch-then-check لأنه بيمنع TOCTOU race conditions.

19ليه عندنا finally في الـtry/catch للأزرار؟

finally بيتنفّذ دايماً — سواء نجح الكود أو رمى error. بنستخدمه عشان نـrestore الـbutton state (disabled = false, textContent = original). من غيره، لو الـrequest فشل، الزر هيفضل disabled للأبد وهيـblock المستخدم. ده مثال على resource cleanup pattern.

20لو لقيت إن الـUI بطيء لما البيانات تكبر، إيه أول خطوة debugging؟

اتّبع التريتيب ده — مش guesswork، methodical:

  1. Profile, don't guess: افتح DevTools → Performance tab واعمل recording.
  2. Network tab: شوف لو فيه request بطيء — ده backend bottleneck.
  3. لو الـbackend بطيء، شغّل db.collection.explain('executionStats') على الـquery — لازم تشوف IXSCAN مش COLLSCAN.
  4. لو الـrender هو البطيء، استخدم virtual scrolling أو pagination.
  5. لو الـDOM فيه آلاف الـnodes، استخدم DocumentFragment أو requestAnimationFrame.

القاعدة الذهبية: "Measure twice, cut once".
